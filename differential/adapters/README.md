# Differential adapters (registry-based)

`hocon-json` adapters for the cross-impl differential harness
(`generate/DifferentialDriver.java`). Each parses + resolves a `.conf` and
prints the resolved tree as canonical JSON to stdout, or a single-line
`{"__error__":{"type":..,"message":..}}` record + exit 3 on failure (exit 2 on
usage error). The driver appends the conf path as the final arg and compares the
result against the Lightbend oracle.

Unlike a sibling-repo checkout, these adapters install their implementation
**from each language's package registry**, pinned to the sibling release, so the
harness reproduces from a clean `xx.hocon` checkout with no sibling repos
present:

| impl | registry | pinned in | adapter |
|------|----------|-----------|---------|
| py   | PyPI (`hocon-parser`)          | `py/requirements.txt` | `py/hocon_json.py` (imports `hocon`) |
| ts   | npm (`@o3co/ts.hocon`)         | `ts/package.json`     | `ts/hocon-json.mjs` |
| rs   | crates.io (`hocon-parser`)     | `rs/Cargo.toml`       | `rs/src/main.rs` (bin depending on the crate) |
| go   | Go module proxy (`o3co/go.hocon`) | `Makefile` `GO_IMPL_VERSION` | `cmd/hocon-json` (installed directly; no vendored source) |

## Build

From the repo root:

```sh
make differential-adapters   # installs all four from their registries
make differential-corpus     # generates the seed corpus (needs a JDK)
make differential            # runs the oracle vs. every adapter
```

Build artifacts (`py/.venv`, `ts/node_modules`, `rs/target`, `go/bin`) are
git-ignored and regenerated by `make differential-adapters`.

## Version bumps

On the next sibling release, bump `GO_IMPL_VERSION` in the `Makefile` and the
pinned version in `py/requirements.txt`, `ts/package.json`, and `rs/Cargo.toml`
together — they must all point at the same release.
