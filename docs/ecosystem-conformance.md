# JVM-free HOCON Ecosystem Conformance

Conformance results for JVM-free HOCON parsers across five languages, measured
against the shared fixtures in this repository. Where
[`compliance-matrix.md`](compliance-matrix.md) tracks the three o3co
implementations at spec-item granularity, this file records the raw
fixture-pass results for the reachable third-party libraries under the same
method.

Each library parses every fixture `.conf` and emits resolved JSON, which is
compared against the fixture's `-expected.json` (generated by Lightbend's
reference implementation). Two corpora are reported:

- **Spec corpus (134)** — the o3co cross-impl fixtures under `testdata/hocon/<group>/`.
- **Lightbend suite (16)** — the upstream Lightbend fixtures that carry a `-expected.json`.

Measured **2026-07-15**. Versions are pinned per row; re-run when a listed
library releases a new version. Libraries are listed alphabetically within each
language.

## Method

One adapter per library — a CLI that reads a `.conf` path and prints the
resolved config as JSON to stdout — is driven over every fixture as an isolated
subprocess, and the output is compared to the fixture's expected JSON. The same
normalization is applied to every library:

- Object key order is ignored; arrays are order-sensitive.
- Numbers compare by value (`1` equals `1.0`); a JSON `string` is not equal to a
  JSON `number`.
- `null`-valued object keys are dropped before comparison, matching HOCON's
  "`null` == missing" semantics and Lightbend's own JSON renderer.
- Each `env-var-list` fixture's `.env` sidecar is loaded into the subprocess
  environment.

Two fixture categories are held outside the reported corpora:

- **Empty-file group (6 fixtures)** — a documented Lightbend-vs-strict
  divergence: Lightbend accepts an empty or comment-only file as `{}`; the o3co
  implementations reject it per spec §L130 ([E10](extra-spec-conventions.md)).
  Libraries differ here by design, so the group is excluded from the score.
- **JVM system-property fixtures** — `test01` and `test03` in the Lightbend
  suite reference `${?java.version}` / `${?user.home}`, which resolve only
  inside a JVM. This is why the Lightbend-suite column tops out at 14/16 for a
  non-JVM parser.

Because the expected JSON is Lightbend-generated and the normalization is
uniform, the numbers measure each library's agreement with the Lightbend
reference implementation.
